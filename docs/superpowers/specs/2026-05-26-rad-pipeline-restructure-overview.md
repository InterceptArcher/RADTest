# RAD Pipeline Restructure — Stakeholder Overview

We are restructuring how RAD generates a prospect briefing in two coupled ways. First, we are replacing the Gamma slideshow service (which has been inconsistent despite three rounds of template work) with a PowerPoint file built deterministically from a master template the sales team controls. Second, we are rebuilding the part of the pipeline that finds executives at the prospect company — switching from the current "pull a bunch of contacts then sort by data completeness" pattern to a surgical, per-role workflow that mirrors how a salesperson would manually research the company. The goal is that every deck always has a CTO, CFO, CIO, and COO slide, each with a real, complete contact record.

Alongside the pipeline changes, we are upgrading the dashboard so that every job is logged centrally (devops can diagnose failures without running anything locally) and so a user can click into a running job and watch each pipeline step happen in real time, with the data filling in stage by stage.

---

## What changes

### The slideshow

| | Today | After v3 |
|--|-------|---------|
| **Service** | Gamma API (cloud-rendered, occasionally inconsistent) | PowerPoint file (`.pptx`) generated server-side from a master template |
| **Output** | "View Slideshow" link on dashboard | "Download Deck (.pptx)" button on dashboard |
| **Reliability** | Each render is a fresh LLM-template binding — visual drift between jobs | Deterministic mechanical fill of a fixed template — every deck looks identical structurally |
| **Where it lives** | Gamma's servers | Supabase Storage (under our control) |

### The contact research

The four stakeholder slides (CTO, CFO, CIO, COO) currently get filled by pulling many contacts from ZoomInfo/Apollo/PDL/Hunter, then sorting by who has the most-complete record. This produces visually consistent decks but often shows the wrong person — someone with great contact details but the wrong title — and silently fails for small companies and rebranded companies.

The new flow mirrors how a salesperson manually researches a company. For each of the four roles, we cascade through tiers of seniority within ZoomInfo (C-suite exact title, then adjacent C-suite titles, then VP, then Director), then move to Apollo, then PDL, and finally fall back to a Claude-powered web search if no structured source surfaces a match. For whichever candidate we pick, we then validate and enrich their record by cross-checking the other sources for any missing fields and looking up their LinkedIn to confirm start date and other details. We move to the next candidate only if a candidate's record cannot be completed.

The dashboard shows all candidates we examined (organized by role bucket), so the user can audit our choices. Each deck still gets exactly four slides, exactly one contact per slide.

### Region filter for international companies

Multinationals (Microsoft, IBM, etc.) typically have global contact records in our data providers, mixing US, EU, and APAC employees. When a salesperson is targeting the Canadian branch specifically, the current system gives them a CTO based in Redmond. We are adding a "Canada only" checkbox on the new-job form; when checked, the contact pipeline will only look at Canadian employees. The general company intelligence (firmographics, news, signals) stays global because that's still relevant context.

### Live progress dashboard

When a job is running, the user can now click into it and see exactly what stage the pipeline is on, what data has been pulled so far, and a live debug log. This replaces today's "running…" black box with full visibility, and it gives devops the data they need to diagnose problems without re-running jobs locally.

### Centralized logging

Every job's full debug trail now persists to Supabase, regardless of which salesperson ran it. If a deck looks wrong or a job fails, the team can pull the log directly from the database.

## What stays the same

- **Template style.** The slides keep the same shape and information density. The master `.pptx` mirrors the current Gamma deck visually.
- **General intelligence.** Firmographics, news, opportunities, signals, account-type bucketing, communication preference — all still generated by the existing LLM Council pipeline.
- **LLM Council.** Still a 20-specialist consensus validating facts. We are removing slide-formatting concerns from its prompt so it focuses on fact accuracy.
- **Sales rep attribution.** Salesperson name still flows through to the deck.
- **Resource recommendations and content matching.** Existing logic (`content_audit.py`) is preserved.

## User-facing improvements

| | Improvement |
|--|-------------|
| Reliability | Every deck has 4 complete stakeholder slides — no more empty C-suite slots or wildly inconsistent visuals |
| Accuracy | Contacts are picked by role-match first, completeness second — no more "great data, wrong title" mistakes |
| Internationals | New region filter solves the multinational-subsidiary problem |
| Transparency | Live progress view + per-job debug log + confidence badge (High / Medium / Low) on every deck |
| Speed of triage | When a job fails, devops can pull the full log directly from Supabase |
| Output portability | Deck is now a `.pptx` file the user owns — works offline, edits in PowerPoint, no Gamma dependency |

## Risks & mitigations

- **Hard cutover.** Gamma is being removed in the same release. If something regresses in the new system, recovery is "revert and redeploy." We are mitigating with a thorough test suite (≥85% coverage, dedicated tests for each red-team failure mode we identified) and golden-file tests for the new PPTX renderer. We considered keeping a feature-flagged rollback path but decided against it to keep the codebase clean — the trade-off is faster maintenance over instant rollback.
- **Cost.** The new contact pipeline uses Claude (Haiku 4.5) for LinkedIn enrichment, ~4 times per job. Estimated per-job cost: $0.15–$0.50 (vs ~$0.10 today). At 100 jobs/day this is ~$15–35/day of additional spend.
- **Latency.** Per-role contact cascade adds a few seconds per role bucket vs the current parallel-spam approach. Expected overall job latency change: +5–15 seconds for a typical company; faster for very large/well-known companies (cache hits) and similar for very small companies (early cascade termination).

## Decisions that were made & locked in

These came out of two brainstorming sessions and are intentionally not re-litigated during implementation:

1. **PowerPoint via `.pptx` master template** — not Google Slides, not a fresh slide template per job, not Gamma v4
2. **Hard cutover** — Gamma code and tests deleted in the same PR, not kept behind a feature flag
3. **Per-role surgical contact pipeline** — not the existing "spam and sort" approach
4. **Canada-only checkbox scopes contacts only** — general company intel stays global
5. **Claude Sonnet 4.6 for the formatting pass; Haiku 4.5 for LinkedIn enrichment** — model choices balance quality vs cost
6. **Stage 3 final-fallback agent** — the only place in the pipeline where Claude orchestrates tool calls; everything else stays procedural for cost and predictability
7. **Live progress dashboard included in this release** — not deferred to a follow-up
8. **Centralized job logging in Supabase** — not a separate logging service
